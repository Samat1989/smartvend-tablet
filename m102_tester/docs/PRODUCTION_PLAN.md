# План доработок до продакшен-готовности

Документ описывает оставшиеся 3 направления работ после направления **«Готов к продакшену»** (kiosk-mode, wake-lock, auto-launch на boot, auto-restart on crash, иконка/название).

Каждое направление можно делать независимо — работаем по приоритету, который задаст реальная эксплуатация.

---

## Чек-лист направления 1 (kiosk-готовность) — что сделано

- [x] `WAKE_LOCK` + `FLAG_KEEP_SCREEN_ON` — экран всегда включён
- [x] `RECEIVE_BOOT_COMPLETED` + `BootReceiver` — авто-запуск после включения
- [x] HOME-категория в Activity intent-filter — можно сделать default-лаунчером
- [x] Sticky immersive (натив + Flutter) — system bars скрыты, swipe не выходит
- [x] `PopScope` в `_Router` — кнопка «назад» не закрывает приложение
- [x] `PopScope` в `DispenseScreen` — нельзя прервать выдачу
- [x] `setShowWhenLocked` + `setTurnScreenOn` — экран включается из standby
- [x] `excludeFromRecents` + `singleTop` + `screenOrientation:portrait`
- [x] `startLockTask()` best-effort на onResume

### Дополнительные улучшения после ревью заводских доков (`c:\m109e\docs\`)

- [x] **Корректная метка кода-результата 2** — было `Underflow`, стало `WireBreak` (по `01_PROTOCOL.md` это «обрыв провода», `ParseM102.m177jx`)
- [x] **Watchdog связи с платой** — `BoardClient.consecutiveFailures` инкрементируется при каждом null-ответе, сбрасывается на любой успешный обмен. Флаг `isHealthy` = `consecutiveFailures < 4` (порог взят из заводского кода: `CreatADH.java:667-717`)
- [x] **Запрос firmware ID при подключении** — `_refreshFirmwareId()` шлёт opcode `0x01` сразу после `port.open()`, кэширует ASCII-версию прошивки и показывает её в шапке сервисного меню. Ранее заводское `m247cxM102` — то же самое
- [x] **Maintenance overlay** на главном экране — когда `BoardClient.isHealthy == false`, поверх каталога показывается полноэкранный знак «Технический перерыв», блокируя добавление товаров в корзину. Тап по overlay по-прежнему считается в скрытый счётчик 5-таппов на сервисный режим
- [x] **Локализованные коды результата мотора** — `Strings.pollResult(code)` мапит 0..10 в `poll_ok`, `poll_overload`, `poll_wire_break`, `poll_timeout`, `poll_curtain_err`, `poll_lock_not_open`, `poll_microswitch` для всех трёх языков. Используется в `DispenseScreen` через новое поле `DispenseStepResult.resultCode`

### Возможные дальнейшие улучшения по дельте с заводской логикой

(Оценено по `c:\m109e\docs\02_LIFECYCLE.md` и `03_PERIPHERALS_AND_CONFIG.md`. Не блокеры — добавляйте по необходимости.)

| Заводская фича | Что мы имеем | Что доделать |
|---|---|---|
| Daily auto-reboot (`RebootService`) | Нет | Добавить Android `AlarmManager` на 03:00 → перезапуск приложения. Чистит память, сбрасывает зависшие state-машины |
| Crash auto-restart (CAOC) | Нет | `Thread.setDefaultUncaughtExceptionHandler` → schedule `MainActivity` через `AlarmManager` + `Process.killProcess`. Без этого падение Flutter оставляет чёрный экран |
| Initial board handshake (`UtilADH.m241`) | Сразу команды без инициализации | Добавить в `BoardClient.connect()`: (1) выключить все DO 0..4, (2) запросить ID, (3) подождать 3 с, (4) применить настройки света/температуры |
| Light-strip schedule (3 окна включения) | Только always-on/off | Добавить в `ClimateConfig.lightSlots: List<({int fromHour, int toHour})>`, проверять в `_applyLight()` |
| Multi-cabinet (sub1/sub2/sub3) | Только slave=1 | Если когда-то понадобятся сдвоенные шкафы — `BoardClient.slaveAddr` уже параметризуется, нужно лишь добавить отдельные poll-циклы и UI для адресов 2/3 |
| Door/lock sensor (DI inputs) | Нет | На M109E в POLL ответе есть биты состояния дверцы; парсить и блокировать выдачу при открытой дверце сейф-лотка |
| Reverse-direction motors (type 0A/0B) | Только 02/03 (forward) | Добавить пункты в `motor_type` UI редактора, для зеркальных аппаратов |
| Police camera / face-pay | Нет | Не нужно — это специфика заводского KZ-приложения |

### Что нужно сделать на самом устройстве (вне кода)

1. **Сделать приложение Home-launcher'ом**: долгое нажатие на «домой» → выбрать «M109E Вендинг» → «Всегда»
2. **Иконка**: положить логотип 1024×1024 в `assets/icon/icon.png` → `dart run flutter_launcher_icons`
3. **Опционально**: для жёсткого kiosk без возможности выйти даже через recents — настроить device-owner через ADB:
   ```
   adb shell dpm set-device-owner kz.smartvend.m102_tester/.DeviceAdminReceiver
   ```
   Это требует factory reset планшета. Без этого `startLockTask()` работает только если оператор вручную закрепил приложение через «недавние».
4. **Отключить уведомления** в настройках Android (звуки, всплывающие)
5. **Подключение к Wi-Fi** настроить заранее, проверить что DNS не блокирует Supabase / SmartVend

## Направление 2: Надёжность платежей и выдачи

**Цель**: автомат не теряет деньги и не подвисает в крайних случаях.

### 2.1 Авто-рефанд при сбое мотора

**Зачем**: сейчас при `dispensed=false` (мотор не выдал) клиент должен звонить владельцу с чеком. Это плохой UX и поддержка.

**Что нужно**:
- API SmartVend для refund — у них есть метод (нужно запросить у них спеку, скорее всего `payment_refund` endpoint аналогично `payment_request`)
- Новый метод `PaymentService.refund({orderid, torderid, amount})` с тем же sign-протоколом
- В `DispenseScreen` после `recordSale` если есть failed items — вызвать refund на сумму = sum(price для не выданных)
- Обработка случая когда refund сам упал (сеть, API даун) — записать в локальную SQLite очередь, доотправить позже

**Файлы**:
- `lib/services/payment_service.dart` — добавить `refund()`
- `lib/services/refund_queue.dart` — новый: SQLite очередь невыполненных рефандов
- `lib/screens/dispense_screen.dart` — вызвать refund после `_recordSale`
- Новая колонка в `sales`: `refund_id text NULL`, `refund_amount numeric NULL`, `refund_status text NULL`

**Оценка**: 4-6 часов работы + согласование refund API со SmartVend.

### 2.2 Очередь на доотправку продаж при потере сети

**Зачем**: если интернет пропал в момент `recordSale`, продажа никуда не записалась, но товар выдан. Деньги ушли клиенту в плюс — нужно фиксировать.

**Что нужно**:
- Локальная SQLite таблица `outbox`: `id, kind, payload_json, created_at, last_attempt, attempts`
- При INSERT в Supabase падает (сеть/HTTP error) — записываем в `outbox`
- Фоновый воркер каждые 30 сек пробует переслать накопленные записи
- Когда восстановилась сеть — все pending уходят в Supabase в порядке создания

**Файлы**:
- `pubspec.yaml` — `sqflite: ^2.x`
- `lib/services/outbox.dart` — новый
- `lib/services/supabase_api.dart` — обернуть `recordSale` в try/save-to-outbox-on-failure

**Оценка**: 6-8 часов.

### 2.3 Ретрай мотора 1 раз перед отказом

**Зачем**: пружина часто срабатывает со второго раза. Сейчас один таймаут = клиенту refund. Дешевле попробовать ещё раз.

**Что нужно**:
- В `BoardClient.dispense()` или `VendingService.dispenseAll()`: если первая попытка вернула `result==Timeout` или `curtain==0` после ok — ждать 1 сек, повторить
- Не повторять при `Overload` (мотор закусило, риск спалить плату)

**Файлы**:
- `lib/board/board_client.dart` — параметр `int retries=1` в `dispense()`

**Оценка**: 1-2 часа.

### 2.4 Локальный кэш каталога (offline-tolerant)

**Зачем**: если Supabase или DNS моргнули — приложение сейчас показывает «Не удалось загрузить товары» вместо последнего известного состояния. Клиент думает что автомат сломан.

**Что нужно**:
- При успешной загрузке `fetchInventory` — сохранить результат в SharedPreferences (или SQLite)
- При падении сети — показать кэшированный каталог + плашку «работаем в офлайн-режиме»
- Очищать кэш при сбросе пэйринга

**Файлы**:
- `lib/services/vending_service.dart` — кэш в `reload()`

**Оценка**: 2-3 часа.

## Направление 3: Опыт владельца

**Цель**: владелец видит что происходит на своих автоматах в реальном времени и может быстро реагировать на проблемы.

### 3.1 Доделать Flutter app `micromarket_app` под вендинг

**Что нужно**:
- Live-фид продаж (Supabase realtime подписка на `sales`)
- Карта моторов 6×6 для каждого автомата с текущими остатками
- Тревоги: «мотор N подряд 3 раза не выдал → возможно неисправен»
- Графики выручки по дням / неделям
- Список аппаратов с агрегатами (выручка сегодня, остатков мало, моторы битые)

**Файлы**:
- `c:\micromarket\micromarket_app\lib\screens\vending_dashboard.dart` — новый
- `c:\micromarket\micromarket_app\lib\services\vending_repo.dart` — Supabase queries

**Оценка**: 2-3 дня.

### 3.2 Workflow пополнения товаров на планшете

**Зачем**: оператор приходит заполнять автомат — сейчас нужно лезть в карточку каждого товара и руками править stock. Долго и легко ошибиться.

**Что нужно**:
- В сервисном режиме новая плитка «Пополнение»
- Список товаров с большими кнопками `+1 / +5 / +10` рядом с каждым
- Кнопка «Принять поставку» сохраняет всё одним батчем + создаёт запись в новой таблице `restocks` (для аудита)

**Файлы**:
- `lib/screens/restock_screen.dart` — новый
- `lib/services/supabase_api.dart` — `bulkUpdateStock([(productId, delta)])`
- SQL миграция: новая таблица `restocks(id, micromarket_id, items jsonb, created_at)`

**Оценка**: 4-6 часов.

### 3.3 Push-уведомления при low-stock и сбоях

**Зачем**: владелец не должен открывать приложение каждые 5 минут чтобы понять что тревожно.

**Что нужно**:
- Supabase Edge Function на cron каждые 30 минут:
  - Сканит `inventory` где `stock <= 2`
  - Сканит `sales_items` за последний час где `dispensed=false` >= 3 раз для одного мотора
- Отправляет push через FCM (Firebase Cloud Messaging) на устройство владельца
- Во Flutter app `micromarket_app` — регистрация FCM токена в `profiles.fcm_token`

**Файлы**:
- `supabase/functions/alert-checker/index.ts` — новый
- Конфиг cron в Supabase
- `c:\micromarket\micromarket_app\lib\services\fcm.dart`

**Оценка**: 1 день.

## Направление 4: Полировка клиентского UX

**Цель**: автомат выглядит и ощущается как премиум-продукт, у клиента остаётся хорошее впечатление.

### 4.1 Анимации добавления в корзину

**Что нужно**:
- При тапе на товар — flying-emoji анимация в иконку корзины
- Цифра в плавающей корзине пульсирует при инкременте
- `flutter_animate: ^4.x`

**Оценка**: 2-3 часа.

### 4.2 Featured product / промо-баннер

**Зачем**: больше визуального drive, можно продвигать новинки и помогать выбору.

**Что нужно**:
- Новая колонка `inventory.is_featured boolean default false` (или таблица `featured_slots`)
- Карусель сверху каталога (3-5 товаров с большими карточками)
- Карусель листается автоматически каждые 5 сек

**Оценка**: 4 часа.

### 4.3 Звуки и тактильная обратная связь

**Что нужно**:
- Бип при успешной выдаче (assets/sounds/success.mp3)
- Низкий тон при отказе мотора
- `HapticFeedback.lightImpact()` при добавлении товара в корзину
- `audioplayers: ^6.x` package

**Оценка**: 2 часа + поиск/запись звуков.

### 4.4 Экран «Заберите товар» с таймером

**После успешной выдачи**:
- Полноэкранный зелёный успех с галочкой
- Большая надпись «Заберите товар из лотка»
- Обратный отсчёт 10 сек, потом возврат на главную
- Если за 10 сек никто не тапнул — всё равно возврат

**Файлы**:
- `lib/screens/dispense_screen.dart` — добавить таймер при `allOk`

**Оценка**: 1-2 часа.

### 4.5 Промо-коды / скидки

**Большая фича** — пока сложно оценить без определения требований. Минимально:
- Поле «Введите промокод» в корзине
- Таблица `promo_codes(code, discount_pct, valid_until, max_uses, used_count)`
- Edge Function валидации кода → возвращает скидку
- Цена в QR пересчитывается с учётом скидки

**Оценка**: 1-2 дня.

## Приоритет (моё мнение)

После того как **Направление 1** реально протестируется на физическом автомате, обычно становится ясно что приоритетнее:

| Сценарий после первого деплоя | Что делать дальше |
|---|---|
| Клиенты ломают UX, теряются | Направление 4 (полировка) |
| Моторы периодически отказывают | Направление 2.1 + 2.3 (refund + retry) |
| Владельцу неудобно следить | Направление 3.1 + 3.3 (dashboard + alerts) |
| Интернет нестабильный | Направление 2.2 + 2.4 (offline) |
| Операторам долго пополнять | Направление 3.2 (restock) |

Без живой эксплуатации приоритет угадывать смысла мало. Поставьте пилотный автомат — соберите боль за неделю — потом возвращайтесь к этому списку.
