## TL;DR — что есть и что нужно сделать

### Что есть
- **Плата M109E** (по протоколу совместима с M102) — управляет всем железом аппарата
- **Заводское приложение** в Android-планшете, которое говорит с платой по UART/USB-Serial
- **Своя плата** для стенда с рабочим TTL-портом + плата из аппарата (только RS485 — TTL отключён аппаратно)
- **Python-скрипт** [`vending_ctrl.py`](../vending_ctrl.py) — реализует базовые команды протокола, проверен на стенде

### Что заменяем
- Заводское Android-приложение (`com.shengma.huishu.hasakesitan`) на своё
- При этом протокол с платой остаётся тот же (это аппаратный API железа, не наш выбор)

### Что НЕ заменяем (и не надо)
- Прошивку самой платы M109E
- Протокол `M102 / RS485 9600 8N1 + Modbus CRC-16 + 20-байтные кадры`

## Что обязательно реализовать в новом приложении

Минимальный набор функций, без которых аппарат не работает как вендинг:

### 1. Слой связи с платой (обязательно)
- Открытие порта (USB-Serial CH340 / `/dev/ttyS*` / прямой COM-порт на ПК)
- 20-байтный фрейм-билдер (addr + func + 16 data + CRC16-Modbus low-first)
- TX-очередь с задержкой ≥ 100 мс между кадрами
- RX-парсер с проверкой длины и CRC
- Цикл опроса (POLL `0x03`) каждые **900 мс** для каждой подключённой платы
- Отслеживание потери связи: 4 пропущенных POLL → флаг "плата в дауне"

### 2. Команды (см. `01_PROTOCOL.md`)
- `0x01` ID — версия прошивки (используется при старте)
- `0x03` POLL — состояние / прогресс выдачи
- `0x05` RUN — запуск мотора (главная команда!)
- `0x07` TEMP — температура
- `0x08` DO — реле (свет, компрессор, нагрев и **силовое питание моторов!**)
- `0x10` HUM — влажность

### 3. Бизнес-логика выдачи
- Каталог "товар → ячейка `HD` → мотор + тип мотора + завеса"
- Команда выдачи: проверить наличие → послать `0x05` → ждать `0x03` со `state=2 (Done)` → разобрать результат
- Многотоварный заказ: **последовательно**, по одному мотору, без параллели
- Обработка ошибок: код результата 0-10 (см. протокол) → решение "успех / неудача / возврат денег"
- Локальный лог транзакций (SQLite)

### 4. Управление микроклиматом
- Чтение температуры каждые 10 с с `addr=01` (главная секция)
- Гистерезис ±4 °C вокруг уставки → включение/выключение компрессора через DO `0x01`
- Защита: задержка 5 минут перед включением компрессора (debounce 30 тиков по 10 с)
- Цикл "работа N мин / отдых M мин" для компрессора
- Опционально: чтение влажности `0x10`, управление подогревом стекла DO `0x02`

### 5. UI
- Сетка товаров с картинками
- Корзина → оплата → подтверждение → анимация выдачи
- Скрытый вход в меню оператора (длинное нажатие 5 с / 5 быстрых тапов)
- Меню оператора: настройки ячеек, цен, каталога, тестов, цены
- Тестовый режим: запуск любого мотора без оплаты

### 6. Платежи (выбрать что нужно)
Минимум один способ. Заводское приложение поддерживает 30+ методов; для запуска достаточно:
- **QR-код** (Kaspi QR / СБП / другой эквайринг) — самый простой, нужен только бэкенд
- **Наличные** (NV-11 + монетоприёмник) — отдельный UART, протокол MDB или прямой
- **NFC карты / банковский терминал** — обычно вендор-специфичный, отдельный API

## Ключевые решения, которые надо принять

### 1. Куда подключать плату
- **Через USB-Serial (CH340) к планшету** — как делает заводское приложение. Минус: нужно физическое подключение, есть глюки с восстановлением драйвера.
- **Через RS485 к Android Box / Raspberry Pi** — гибче, но добавляет хост.
- **Через USB-RS485 к ПК** — идеально для разработки и для аппарата на выезде, если использовать ПК как контроллер.

> **Рекомендация:** на планшет через USB-Serial для совместимости с существующим железом.

### 2. Какой бэкенд
- **Без бэкенда** (всё в SQLite на устройстве) — самый быстрый старт, но никакого удалённого мониторинга/OTA
- **Свой бэкенд** (Spring Boot / FastAPI / Node.js) — полный контроль, нужно его сделать
- **Готовый MDM** (e.g. ESPER, MOKO) — для управления устройствами без вендинговой логики

> **Рекомендация:** начать без бэкенда, добавить позже простой REST для синхронизации остатков и цен.

### 3. Платежи
В Казахстане: **Kaspi QR** (есть Android SDK), **Halyk QR**, **CT KZ**. Также можно SDK мобильных эквайеров.
> **Рекомендация:** Kaspi QR как первый вариант — самый распространённый в KZ.

### 4. Совместимость с заводским облаком
- Если хотите оставить связь с заводским сервером (`ff.smshj.com`) — нужно реализовать TCP-протокол с фреймами `B000`/`A001`/`01710022` (см. `02_LIFECYCLE.md` секцию B.2)
- **Скорее всего, не нужно** — лучше отвязаться от китайского сервера, иначе зачем менять приложение

> **Рекомендация:** игнорировать заводское облако, делать со своей серверной стороной (если она нужна).

## Критические "грабли" — на что обратить внимание

1. **Силовое питание моторов через DO** — на M109E одно из реле включает 24В на шину моторов. Без него команда `0x05 RUN` уйдёт впустую. **При старте приложения нужно включить нужное реле.** На стенде BS с этим обычно проблем нет (питание моторов всегда включено), а на боевом — может быть отдельный канал.

2. **TTL vs RS485** — на боевой плате (которую вытащили из аппарата) транзистор TTL-порта аппаратно отключён. Подключение через TTL не сработает. Обязательно RS485.

3. **Адрес платы по умолчанию = 1**, но в аппарате может быть другой (если несколько секций). Сделать `--scan-addr` через адреса 1-8.

4. **Индексы моторов 0-based в команде**, а на корпусе/проводке часто 1-based. То есть L1 на проводке = motor 0 в команде.

5. **CRC16 — Modbus, low-byte-first на проводе.** Стандартная ошибка — поменять байты местами и часами не понимать почему плата не отвечает.

6. **20-байтные кадры строго.** Плата отбрасывает всё, что не 20 байт. Не сокращайте до 18 если данных мало — заполняйте нулями.

7. **Задержка 100 мс минимум между кадрами.** Без неё плата начинает их склеивать и теряет.

8. **При зависании платы** — заводское приложение делает `restartCOM()` и в крайнем случае ребут планшета через 5 минут. Реализуйте watchdog.

9. **Первичная инициализация M102 (по UtilADH.m241)**:
   1. Закрыть все DO 0-4 на всех станциях (`0x08` с state=00)
   2. Запросить ID `0x01` каждой подключённой станции
   3. Через 3 с — настроить свет и температуру

10. **Питание планшета и платы должно иметь общую землю** — без этого RS485 может работать нестабильно.

## Переход от заводского — миграция данных

Заводская SQLite БД лежит на устройстве в `data/data/com.shengma.huishu.hasakesitan/databases/vendingMachine.db`. Можно:

1. Снять её через ADB: `adb shell run-as com.shengma.huishu.hasakesitan cat databases/vendingMachine.db > backup.db`
2. Импортировать таблицу `GOOD_ENTITY` в новую Room-БД, преобразовать поля:
   - `HD` (4 hex) → `address` (1 byte) + `motor_index` (1 byte)
   - `PRICE` (cents в hex) → `Long priceCents`
   - `RL_HD` → массив дополнительных моторов для сдвоенных ячеек
3. Изображения товаров лежат в `goodsImg/` папке устройства — можно перенести.

## Полезные команды для разработки

```bash
# Сканировать адреса плат
python vending_ctrl.py --port COM6 --scan-addr

# Получить ID и температуру
python vending_ctrl.py --port COM6 --info

# Перебрать все DO
for i in 0 1 2 3 4; do
    python vending_ctrl.py --port COM6 --do $i 1
    sleep 1
done

# Запустить мотор 5 (физическая ячейка L6 если 1-based)
python vending_ctrl.py --port COM6 --motor 5 --type 2

# Сканировать наличие моторов
python vending_ctrl.py --port COM6 --scan-all
```
